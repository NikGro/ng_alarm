## Why NG Alarm?

The existing alarm solutions for Home Assistant are good — but they weren't quite right for my setup. I needed something with more surgical control over *which* sensors matter *when*, without the alarm screaming at me because my robot vacuum triggered a motion sensor at 2am while the alarm was armed.

Specifically, the deal-breakers with existing options were:

- **No per-zone sensor bypass rules** — I needed the vacuum to be able to trigger sensors in certain zones while the alarm stays armed in others.
- **No script-based action flows** — I wanted full control over *what happens* on a state transition, not just notifications.
- **Config UI that felt bolted on** — I wanted something that looked and behaved like a first-party HA integration.

So instead of forking an existing integration and trying to navigate an unfamiliar codebase, I spec'd out exactly what I wanted and built it from scratch with AI assistance. The result is NG Alarm.

## Features

### Zone Management

NG Alarm uses a **zone-based model** rather than treating the entire home as a single alarm boundary. Each zone is independently configurable with its own sensors, delays, bypass strategy, and behavior.

- Create any number of zones with custom names and icons
- Each zone can be individually armed or contribute to the overall alarm state
- Zone-level entry/exit delays are independently configurable
- Zones can have different behavior per arm mode (e.g. the garage might be included in *Away* but not *Night*)

### Arm Modes

Four standard arm modes are supported, each of which can activate a different set of zones and sensor rules:

| Mode | Typical Use Case |
|---|---|
| **Away** | Full perimeter active, no presence expected |
| **Home** | Interior zones relaxed, perimeter sensors active |
| **Night** | Selective sensors active for overnight presence |
| **Vacation** | Extended-absence profile with maximum coverage |

Each arm mode can be triggered manually from the UI, via the alarm control panel entity, or programmatically via scripts and automations.

### Sensor Rules & Bypass

This is the feature that started the whole project. Sensor rules let you define exactly which sensors are active in which zones and under which conditions — with three bypass strategies available:

| Bypass Strategy | Behavior |
|---|---|
| **None** | Sensor is always active when the zone is armed |
| **Entity State** | Sensor is bypassed when a specified entity is in a given state (e.g. bypass motion sensors when `vacuum.roomba` is `cleaning`) |
| **Template** | Full Jinja2 template for complex bypass conditions |

Bypass rules are evaluated per zone, so the same physical sensor can be active in one zone and bypassed in another simultaneously.

### User & Code Management

NG Alarm supports multiple users, each with their own PIN code and permission level. This allows you to give family members, housekeepers, or other trusted people the ability to arm/disarm without sharing a single global code.

- Add and remove users via the configuration panel
- Assign individual PIN codes per user
- Control permissions per user (e.g. arm only, arm + disarm, full access)
- Disarm events are attributed to the specific user who triggered them — visible in the event log and passed to action scripts

> 📸 **Screenshot placeholder** — User management panel

### Action Builder

The action builder lets you define automation-like rules that fire scripts when specific state transitions occur. Unlike standard HA automations, these rules are tightly integrated with the alarm state machine and can filter on:

- **From state** — the alarm state before the transition
- **To state** — the alarm state after the transition
- **Through state** — intermediate states visited during the transition
- **By user** — only trigger if a specific user caused the transition

Each rule targets one or more HA scripts, which are called with a rich set of [variables](#action-variables-reference). This means you can do things like:
- Play a different TTS announcement depending on *who* disarmed the alarm
- Flash lights red only when triggered by a specific zone
- Send a push notification with the exact sensor name that tripped the alarm

> 📸 **Screenshot placeholder** — Action builder rule editor

### Event Log

Every alarm event is written to a persistent event log accessible from the sidebar panel. The log captures state transitions, the actor responsible, and the triggering sensor where applicable.

- View all events directly in the panel
- Clear the log via button or REST API
- Export log data for external processing
- Optionally expose the log as a Home Assistant sensor entity for use in dashboards or automations

> 📸 **Screenshot placeholder** — Event log panel

### Configuration UI

The entire integration is configured through a sidebar panel built with native Home Assistant frontend components. No YAML required (beyond the initial integration setup).

The UI uses standard HA selectors throughout — entity pickers, state selectors, icon pickers — so it feels like a first-party experience rather than a bolted-on custom frontend.

## Installation

### Via HACS (Recommended)

1. Open HACS in your Home Assistant instance
2. Go to **Integrations** → click the three-dot menu → **Custom repositories**
3. Add the following URL with category **Integration**:
   ```
   https://github.com/NikGro/ng_alarm
   ```
4. Find **NG Alarm** in the HACS integration list and install it
5. **Restart Home Assistant**

### Manual Installation

1. Download or clone this repository
2. Copy the `custom_components/ng_alarm` directory into your HA configuration directory:
   ```
   config/
   └── custom_components/
       └── ng_alarm/   ← copy here
   ```
3. Restart Home Assistant

---

## Setup

After installation and restart:

1. Go to **Settings → Devices & Services → Add Integration**
2. Search for and select **Alarm**
3. Complete the configuration flow
4. The **Alarm** entry will appear in your HA sidebar — open it to configure zones, sensors, users, and action rules

## REST API

The integration exposes a set of HTTP endpoints that can be used for external automation, dashboards, or scripting. All endpoints require a valid HA long-lived access token.

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/api/ng_alarm/config` | Retrieve the full current configuration as JSON |
| `POST` | `/api/ng_alarm/config` | Update the configuration |
| `POST` | `/api/ng_alarm/reload` | Reload the integration without restarting HA |
| `GET` | `/api/ng_alarm/events` | Retrieve the full event log as JSON |
| `POST` | `/api/ng_alarm/events/clear` | Clear the event log |

**Example — retrieve config:**
```bash
curl -X GET http://homeassistant.local:8123/api/ng_alarm/config \
  -H "Authorization: Bearer YOUR_LONG_LIVED_TOKEN" \
  -H "Content-Type: application/json"
```

## Action Variables Reference

When an action rule fires, every configured script target is called with the following variables available in the script context:

| Variable | Type | Description |
|---|---|---|
| `from_state` | `string` | The alarm state immediately before the transition |
| `to_state` | `string` | The alarm state immediately after the transition |
| `alarm_state` | `string` | The current alarm state at time of script execution |
| `alarm_mode` | `string` | The active arm mode (`away`, `home`, `night`, `vacation`) |
| `actor` | `string` | The user or entity ID that triggered the transition |
| `triggered_sensor` | `string` | Entity ID of the sensor that triggered the alarm (if applicable) |
| `triggered_sensor_name` | `string` | Friendly name of the triggering sensor (if applicable) |

**Example usage in a script:**
```yaml
- service: notify.mobile_app_phone
  data:
    title: "Alarm triggered"
    message: "{{ triggered_sensor_name }} in zone tripped the alarm."
```
